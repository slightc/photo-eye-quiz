# 摄影审美对比测验 · 出题指南

## 这是什么

一个纯前端的中文小测验:每题给出 **一对取景器画面**(A 和 B),让玩家选「构图 / 用光 / 色彩更好的那一张」。选完即时判对错,弹出解析卡;答完一卷可逐题回看。

画面 **不是真实照片,而是 SVG 矢量示意图** —— 这是刻意的设计:

- **可控**:好图与坏图之间只差「一个审美变量」,其它全相同,玩家能干净地看出差别。
- **无版权**:全部矢量手绘,不依赖任何照片素材。
- **轻量**:单 HTML 文件,无外部依赖。

出题的本质,就是 **挑一条摄影审美原则,造一对「只在这条原则上不同」的画面**,再把判断和理由讲清楚。

## 数据流:JSON 是唯一数据源

```
questions/<order>-<id>.json   ←  题库唯一数据源(每题一个文件)
        │
        ├─ questions/index.json   ←  清单(自动生成,勿手改)
        │
   node build.mjs                 ←  把所有 JSON 回灌进 HTML
        │
        ▼
index.html                        ←  成品(RAW 数组在两个标记注释之间被整体替换)
```

**改题流程永远是:改 JSON → 跑 `node build.mjs` → 完成。**
不要直接手改 HTML 里的 `const RAW = [...]`,它会被下次 build 覆盖。

新增一题:在 `questions/` 下新建 `NN-yourid.json`(`NN` = 两位序号,决定出场顺序),然后 `node build.mjs`。建议同时手动把它登记进 `index.json`,或重跑生成脚本。

## JSON Schema

每个题目文件长这样(以 01-level.json 为例):

```json
{
  "id": "level",
  "order": 1,
  "direction": "取景",
  "category": "取景",
  "subcategory": "水平",
  "en": "LEVEL",
  "difficulty": "入门",
  "tags": ["水平线", "端平", "风光", "海景", "基础", "新手必看"],
  "title": "哪一张更稳？",
  "hint": "同一片落日海面,只差一个动作。",
  "svg": {
    "good": "<rect .../>${grain}",
    "bad":  "<rect .../>${grain}"
  },
  "verdict": "更稳的是地平线水平的一张。",
  "points": [
    "风光里歪斜的海/地平线会让整幅画失衡……",
    "要倾斜就得倾斜得彻底——大角度的「荷兰角」才成立……"
  ],
  "soft": null
}
```

字段逐条说明:

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 英文短标识,题库内唯一,文件名也用它。如 `thirds`、`dof`。 |
| `order` | number | 出场序号,和文件名前缀一致,决定题目顺序。 |
| `direction` | string | **方向**,归一化的 8 个大类之一(见下)。 |
| `category` / `subcategory` | string | 显示在题面的「眉标」,渲染成 `category · subcategory`。category 通常等于 direction,但可更口语(如 `用光取景`)。 |
| `en` | string | 全大写英文小标签,纯装饰(如 `LEVEL`、`RULE OF THIRDS`)。 |
| `difficulty` | string | **难度**:`入门` / `进阶` / `高阶` 三选一(见下)。 |
| `tags` | string[] | **标签**,6 个左右,用于检索/归类(见下)。 |
| `title` | string | 题面主问句,口语化,带问号。如「哪张人像更立体？」 |
| `hint` | string | 一句副提示,点出「两张只差在哪个维度」,但不直接给答案。 |
| `svg.good` / `svg.bad` | string | **好图 / 坏图的内层 SVG 模板串**(见「画面怎么画」)。 |
| `verdict` | string | 解析卡标题,一句话点明哪张更好、好在哪。 |
| `points` | string[] | 解析要点,通常 2 条:一条讲好图为何好,一条讲坏图为何弱(或:一条原则 + 一条提醒)。 |
| `soft` | object\|null | 「软话」补充,缓和「绝对化」语气。结构 `{ label, text }`,`label` ∈ {`并非定律`, `练法`, `补充`};无则 `null`。 |

> **关于 `soft`**:摄影审美很少有铁律。当一条原则有明显例外、或想给玩家一个可操作的练习时,用 soft 补一句。
> - `并非定律` —— 点出例外:「平光也有用处(高调、美妆);但想要立体的人像,侧光几乎总更好。」
> - `练法` —— 给个可练的动作:「下次拍跳跃,连拍 + 提前半秒按,挑最高点那张。」
> - `补充` —— 其它中性补充。

## 三个维度:难度 / 方向 / 标签

### 难度(difficulty)三级

判据是「**要看出这个差别,需要多少眼力积累**」:

| 难度 | 标准 | 例子 |
|---|---|---|
| **入门** | 一旦指出来,任何人立刻就懂的基本功。歪没歪、挤没挤、堵没堵。 | 地平线端平、三分法、头顶留白、靠近主体填满画面、天空别占太多。 |
| **进阶** | 需要刻意练习才会养成的「眼力」。要主动去看层次、平衡、引导、背景。 | 负空间、前景框架、对角线动感、视觉平衡、背景净化、避免相撞。 |
| **高阶** | 用光、时机、色温层面的微妙判断,差别细、靠经验。 | 伦勃朗光的光位、白平衡冷暖、高调/低调影调、决定性瞬间、追拍慢门、掠射光质感。 |

一卷题目最好 **入门铺路、进阶为主、高阶点睛**,由易到难大致递进(参考现有题库:入门 9 / 进阶 23 / 高阶 6)。

### 方向(direction)8 类

归一化的大方向,**新题尽量落进这 8 类之一**,不要新造:

| 方向 | 管什么 |
|---|---|
| **构图** | 元素在画面里怎么摆:三分法、引导线、负空间、对称、平衡、三角形、奇数法则…… |
| **取景** | 框多大、框哪里、裁哪里:水平、头顶空间、关节裁切、天空占比、做减法、避免相撞。 |
| **用光** | 光从哪来、光的质感:黄金时刻、侧光、剪影、伦勃朗光、掠射质感。 |
| **色彩** | 颜色关系:色彩和谐、点缀色跳色、白平衡冷暖。 |
| **影调** | 明暗与调性:明暗对比、高调/低调。 |
| **镜头** | 镜头/快门带来的效果:景深虚化、追拍慢门。 |
| **视角** | 机位高低、拍摄角度:低机位仰拍。 |
| **时机** | 按下快门的瞬间:决定性瞬间。 |

> `category` 可以比 `direction` 更具体或更口语,但 `direction` 必须是上表 8 选 1,这样 index 才能稳定聚合。

### 标签(tags)

每题 **6 个左右**,混合三种粒度,方便不同角度检索:

- **术语**:`三分法`、`负空间`、`伦勃朗光`、`panning`(可中英)。
- **题材**:`风光`、`人像`、`海景`、`街拍`、`建筑`。
- **场景/受众**:`新手必看`、`基础`、`氛围`、`耐看`。

## 画面怎么画(技巧核心)

### 渲染契约(务必遵守)

- `svg.good` / `svg.bad` 存的是 **内层 SVG 片段**(不含外层 `<svg>`),运行时作为 **模板字符串** 在宿主 HTML 的 `<script>` 作用域里求值。
  - 因此你可以、也应该在里面写 `${grain}`、`${person(120, 300, 1)}` 这样的 **helper 调用** 和插值。
  - 求值后由 `svgWrap(inner)` 包成:`<svg viewBox="0 0 600 400" preserveAspectRatio="xMidYMid slice">…</svg>`。
- **画布坐标系固定 600 × 400**,左上角为原点,y 向下增大。地面/海平面通常在 y≈260–300。
- **渐变和滤镜由宿主 `<defs>` 提供**,用 `fill="url(#skyDusk)"`、`filter="url(#dof)"` 引用,不要自己重定义。
- 模板串里 **用双引号** 作 SVG 属性引号(`fill="..."`),**不要用反引号**(反引号会和外层模板串冲突)。
- 收尾几乎总是拼一个 `${grain}` 加颗粒质感(它属于取景框架,不算「题目内容差异」,好坏图都加)。

JSON 里这些是 **字符串**,所以 `${...}` 以字面文本存储,由 build.mjs 原样放进 HTML 后才真正求值 —— 你在 JSON 里照常写 `${grain}` 即可。

### helper 速查

宿主作用域内可直接调用(坐标都在 600×400 系内,`s` 为缩放系数):

| helper | 签名 | 画什么 |
|---|---|---|
| `grain` | 常量字符串 | 颗粒覆盖层,直接 `${grain}` 拼上。 |
| `person` | `person(x, ground, s)` | 站立人物剪影。`ground`=脚底 y(站立基线),**朝右**。 |
| `bust` | `bust(cx, topY, s)` | 头肩半身剪影。`cx`=中心 x,`topY`=头顶 y。 |
| `gull` | `gull(x, y, s)` | 海鸥(M 形轮廓),`(x,y)`=中心。 |
| `boat` | `boat(x, y, s)` | 小帆船,`(x,y)`=船底中心。 |
| `tree` | `tree(x, ground, s, trunk, crown, lit)` | 蓬松树。`trunk`/`crown`=树干/树冠色,`lit`=高光描边色(传空串 `''` 则不打光)。 |
| `ridge` | `ridge(y, h, fill, seed)` | 远山山脊层。`y`=山顶基准线,`h`=起伏高度,`seed`=种子(同 seed 同形状)。 |
| `flower` | `flower(x, y, s, petal, center)` | 六瓣花。`(x,y)`=花心,`petal`/`center`=花瓣/花心色。 |
| `car` | `car(x, ground, s)` | 汽车剪影,`ground`=轮子着地 y。 |

> 没有合适 helper 时,直接手写 `<rect>`、`<circle>`、`<path>`、`<polygon>`、`<line>`、`<g transform=...>` 即可。

### 可用渐变 / 滤镜(来自宿主 `<defs>`)

**线性渐变**(`fill="url(#…)"`):

| id | 用途 |
|---|---|
| `skyDusk` / `seaDusk` | 黄昏天空 / 海面(暖) |
| `skyNoon` | 正午晴空(冷蓝) |
| `field` | 草地/田野(绿) |
| `mist` | 雾气/远景柔化(浅灰) |
| `mountSky` | 山景天空 |
| `portraitSky` | 人像背景天空 |
| `goldGround` | 金色地面/逆光地表 |

**滤镜**(`filter="url(#…)"`):

| id | 效果 | 典型用法 |
|---|---|---|
| `grain` | 胶片颗粒 | 已封装进 `grain` 常量,一般不直接用。 |
| `dof` | 高斯模糊 | 背景虚化、浅景深的「坏图清晰 / 好图虚化背景」对比。 |
| `motion` | 水平方向运动模糊 | 追拍(panning)、动感拖影。 |

### 「成对只改一个变量」原则 ⭐

这是出好题的命门:**good 和 bad 应当尽量只在「这道题考的那条原则」上不同,其余构成元素(主体位置、配色、天空、颗粒……)保持一致。**

- 考「地平线水平」→ 两张同一片海、同一个落日,只有 bad 用 `transform="rotate(-6 …)"` 把海面歪了。
- 考「背景虚化」→ 两张同一个人,只有 good 给背景套 `filter="url(#dof)"`。
- 考「头顶空间」→ 两张同一个半身像,只有 bad 把 `bust` 的 `topY` 顶到画面最上沿。

差异越干净,题目越「公平」,玩家越能学到那一条原则。**切忌一次改好几处**,否则玩家不知道到底在比什么。

## 文案写作语气

整体口吻:**像一个懂行又不端着的朋友在帮你看片**。讲人话,少术语堆砌,术语出现时顺带用大白话解释。

- **title** —— 口语问句,带悬念:「哪张更稳？」「哪张人像更立体？」别写成「关于水平线的测验」。
- **hint** —— 点出「只差在哪个维度」,但不剧透答案:「看光从哪个方向打在脸上。」
- **verdict** —— 一句话给结论 + 一句话原因:「更立体的是脸上有侧光、有明暗过渡的一张。」
- **points** —— 通常 2 条,**一条夸好图、一条点坏图**(或:一条讲原则、一条提醒边界)。具体、可感,讲清「为什么观感不同」,而不是空喊「构图更好」。
- **soft** —— 给绝对化的判断留个口子或给个练法(见上)。

## 出一道新题 · checklist

1. **选一条原则**,归到某个 `direction`,定 `difficulty`。
2. **构思一对画面**:同一个场景,只在这条原则上拉开差距(good vs bad)。
3. 用 helper + 渐变 + 手写 SVG 画出 `svg.good` / `svg.bad`,**两张除目标差异外尽量一致**,各自 `${grain}` 收尾。
4. 写 `title` / `hint` / `verdict` / `points`,必要时加 `soft`。
5. 起 `id`、配 6 个左右 `tags`、定 `order`、填 `category`/`subcategory`/`en`。
6. 存成 `questions/NN-id.json`,登记进 `index.json`。
7. **`node build.mjs`** 回灌,在浏览器里打开 `index.html` 自测:
   - good/bad 两张确实 **只差目标变量**;
   - 颗粒、渐变正常渲染,无 SVG 报错;
   - 文案读起来顺、结论无歧义。

### 常见坑

- ❌ 在 SVG 模板里用反引号当属性引号 → 和外层模板串打架。**用双引号。**
- ❌ 一题改了好几个变量 → 玩家不知道在比什么。**只改一个。**
- ❌ 自造 `direction` → index 聚合乱掉。**落进现有 8 类。**
- ❌ 直接手改 HTML 的 RAW 数组 → 下次 build 被覆盖。**只改 JSON。**
- ❌ 渐变/滤镜 id 写错(大小写、拼写)→ 画面变纯黑/无效果。**对照上表。**
- ❌ verdict/points 只说「更好」却不说「为什么」→ 玩家学不到东西。**讲清观感差异。**
